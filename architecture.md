## Overview

The system is a voice-first interview practice agent. A candidate speaks; the agent transcribes, evaluates against a fixed reference dataset, responds as a human interviewer, and returns a spoken reply. All in under 3 seconds end-to-end on a warm path.

The three hardest engineering problems were: (1) keeping the LLM grounded without leaking the ideal answer; (2) making follow-up logic adaptive but bounded; and (3) hitting acceptable latency with four sequential API calls.

---

## 1. Retrieval Design

### How the Q&A is stored and chunked

Each entry in `data/qa_dataset.json` is treated as a **single logical chunk**: the question and its ideal answer are concatenated into one string before embedding:

```
"Question: <text>\nIdeal answer: <text>"
```

**Why not split them?** If question and answer were separate vectors, retrieval on the candidate's answer could return only the question embedding — which gives no reference context to the evaluator. Keeping them together means every FAISS hit is a complete grounding unit: the question, the ideal answer, the keywords, and the difficulty level.

FAISS stores `text-embedding-3-large` vectors (3072 dims) with inner-product search after L2-normalisation, which is equivalent to cosine similarity. The index is built once at startup and serialised to `data/faiss_index.index` + `.meta`. Restarts reload from disk without re-calling the Embeddings API.

### Two-mode retrieval: semantic for context, exact for scoring

There are two distinct retrieval moments:

**1. Semantic retrieval (context enrichment):**
After the candidate speaks, we embed their answer and run `faiss.search(top_k=3)`. The top-k results give the LLM interviewer additional related context — useful when the candidate's answer drifts into adjacent territory (e.g. an async Python answer might retrieve both the concurrency question and the GIL question).

**2. Exact lookup (scoring):**
Once a question is assigned, we use `get_by_id(question_id)` — a direct dictionary lookup, not a vector search. This guarantees we score the candidate against exactly the right reference, regardless of how similar their answer was to other Q&A pairs.

This two-mode approach is the key architectural insight: **semantic search for context, deterministic lookup for evaluation**.

## 2. Keeping the LLM Behaving Like an Interviewer

### The information firewall

Evaluation and interviewer response are **two separate LLM calls** with completely different prompts:

```
EVALUATION_PROMPT  →  private, includes ideal_answer  →  returns JSON score
INTERVIEWER_PROMPT →  only receives gap labels (e.g. "low technical_depth")
                   →  never sees ideal_answer text
```

The interviewer prompt only knows which dimensions scored low. It cannot say "the ideal answer mentions Redis" because it was never told that. It can only say "Can you elaborate on the caching layer?" — which is exactly what a real interviewer would do.

The one exception: if `overall < 4.0` AND `followup_count >= 2` (candidate is clearly stuck after two follow-ups), the system prompt permits a 1–2 sentence conceptual hint. Even then, the hint is generated by the LLM from the gap labels, not copied from the ideal answer.

### Follow-up logic: adaptive but bounded

The `decide_followup` node in LangGraph implements the rule:

```python
if is_weak and followup_count < 2 and candidate_answer:
    # generate follow-up targeting lowest-scoring dimension
else:
    # advance to next question regardless
```

The cap of 2 follow-ups per question is critical. Without it, a genuinely difficult question could stall the entire interview. The agent moves on after two attempts — reflecting real interviewer behaviour.

The follow-up prompt (`FOLLOWUP_PROMPT`) identifies the lowest-scoring dimension from the score dict and generates a question targeting that specific gap. This means follow-ups are always targeted, never generic.

### Question ordering

A third LLM call (`NEXT_QUESTION_PROMPT`) selects the next question from the remaining pool by considering the last question asked and the conversation flow. This produces a more coherent interview arc than random ordering — for example, it will naturally follow a system design question with a scalability follow-up question rather than jumping to a behavioural question.

## 3. Latency

### Where the time goes (warm path, sequential)

| Step | Typical latency |
|---|---|
| STT — Whisper API, 10–20s audio | 800–1200 ms |
| Embed candidate answer (text-embedding-3-large) | 200–400 ms |
| FAISS search (local, in-memory) | < 5 ms |
| LLM evaluation (GPT-4o, JSON mode, ~400 tok) | 600–1200 ms |
| LLM interviewer response (~150 tok) | 400–800 ms |
| TTS — ElevenLabs, ~30 words, first byte | 300–600 ms |
| **Total** | **~2.3–4.2 s** |

### Optimisations implemented

**1. Redis embedding cache.** `embed_text()` hashes the input text (MD5) and caches the 3072-dim vector in Redis with a 1-hour TTL. Repeated phrases (interview openings, common follow-up starters) are served in < 1 ms instead of 200–400 ms.

**2. FAISS on disk.** The index is built once (on first boot) and reloaded from disk on subsequent restarts. Zero rebuild cost in production.

**3. TTS streaming.** ElevenLabs' `/stream` endpoint returns MP3 chunks as they're synthesised. The frontend begins audio playback on the first chunk — the user hears the start of the response while synthesis is still in progress. Effective perceived latency is reduced by ~500 ms.

**4. Async throughout.** Every API call uses `async/await`. The FastAPI backend handles multiple concurrent sessions without blocking.

**5. LLM call separation.** The evaluation call and the interviewer call are sequential (the interviewer needs the evaluation result to know whether to follow up). However, the FAISS search runs concurrently with the exact `get_by_id()` lookup using `asyncio.gather()`.

### Biggest remaining bottleneck

STT (Whisper) accounts for 30–40% of total latency and is hard to optimise without changing provider. Options:
- **Deepgram Nova-2**: streaming STT with partial transcripts available in ~200 ms, vs Whisper's batch model. Would cut total latency by ~700 ms.
- **Client-side VAD**: detect end-of-speech in the browser and send audio immediately rather than waiting for the user to click "submit".
- **Local Whisper on GPU**: a `large-v3` model on an A10G GPU processes 30s of audio in ~400 ms, faster than the API with no network round-trip.

## 4. Dataset Updatability

The Q&A dataset (`data/qa_dataset.json`) is a plain JSON file. Adding, editing, or removing a question requires:

1. Edit the JSON file.
2. Delete the cached FAISS index (`data/faiss_index.index`, `data/faiss_index.meta`).
3. Restart the backend — it rebuilds and re-caches the index automatically.

No code changes. No schema migrations. No redeployment of application logic. The entire pipeline is parameterised by the dataset structure.

## 5. Why These Specific Technology Choices

| Decision | Alternative considered | Reason for choice |
|---|---|---|
| FAISS (local) | Pinecone, Weaviate | 10–50 Q&A records don't justify managed vector DB ops cost or latency |
| LangGraph | Plain async functions | Typed state machine makes the flow inspectable, testable, and easy to extend with new nodes |
| Redis for session state | In-memory dict | Backend must be horizontally scalable; in-memory state breaks with multiple instances |
| ElevenLabs multilingual v2 | Azure TTS per language | Single model handles EN/HI/DE with natural prosody; avoids per-language voice management |
| text-embedding-3-large | text-embedding-ada-002 | 3072-dim vectors give significantly better cosine similarity for technical text retrieval |
| GPT-4o | GPT-4-turbo, Claude 3 | Best JSON mode reliability; lower hallucination rate on technical evaluation tasks |
